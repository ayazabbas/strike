# Strike v1.1 Smart Contract Security Audit Report

**Date:** 2026-03-22
**Auditor:** Claude Opus 4.6 (Automated)
**Scope:** All contracts in `src/` (10 files, ~2,500 lines Solidity)
**Commit:** `ae11599` (sdk: update ABIs and types for v1.1 contracts)
**Framework:** Foundry, Solidity ^0.8.25, OpenZeppelin 5.x

---

## Executive Summary

Strike v1.1 is a binary outcome prediction market CLOB with Frequent Batch Auctions on BNB Chain. The v1.1 release adds internal position tracking, gas optimizations with unchecked arithmetic, time-based finality with challenge mechanism, and pool insolvency fixes from a prior audit.

The codebase is mature with 314 test functions and good coverage of primary flows. Prior audit findings (C-01 pool insolvency, H-02 finality) have been addressed. However, the audit identified **1 High**, **3 Medium**, and **3 Low** severity findings, primarily around cross-contract reentrancy in ERC1155 markets, oracle edge cases, and chunked settlement correctness.

**Overall Risk Assessment: MODERATE.** No critical fund-theft vulnerabilities found. The High finding is a DoS vector on ERC1155 markets that can be mitigated by using internal positions. Core settlement math and access control are sound.

## Findings

### H-01: Cross-Contract Reentrancy via ERC1155 Callbacks Can DoS Batch Clearing

**Severity:** High
**Contract:** BatchAuction.sol, OrderBook.sol
**Functions:** `_settleBuyOrder`, `_settleSellOrder`, `cancelOrder`

**Description:**

During batch settlement, ERC1155 token operations (`mintSingle` for buy fills, `transferEscrowTokens` for sell order returns) trigger `onERC1155Received` callbacks on the recipient address. If the recipient is a smart contract, it can reenter `OrderBook` during the callback.

`BatchAuction.clearBatch` holds BatchAuction's `nonReentrant` lock, but **OrderBook's reentrancy guard is not locked**. After `vault.settleFill()` returns (releasing Vault's lock), `outcomeToken.mintSingle()` is called. The recipient's callback can then call `OrderBook.cancelOrder()`, which succeeds because neither OrderBook's nor Vault's guards are locked at that point.

**Attack Sequence:**

1. Attacker deploys a malicious contract that implements `onERC1155Received`.
2. Attacker places two orders in the same batch on an ERC1155 market (e.g., two Bid orders).
3. `clearBatch` is called. During settlement of the first order, `mintSingle` triggers the attacker's callback.
4. Inside `onERC1155Received`, the attacker calls `orderBook.cancelOrder(secondOrderId)`.
5. The cancel succeeds: second order's `lots` is zeroed in storage, collateral refunded.
6. Settlement continues. When it reaches the second order, `orderBook.reduceOrderLots()` reverts because `orders[secondOrderId].lots = 0 < lotsToReduce`.
7. The entire `clearBatch` transaction reverts.
8. This can be repeated indefinitely, permanently blocking batch clearing for the market.

**Impact:**

Permanent DoS on batch clearing for any ERC1155 market. All users' orders and collateral are stuck until market expiry + 24h (when `cancelExpiredOrders` can be called). Affects only ERC1155 markets — internal position markets are immune (no ERC1155 callbacks).

**Proof of Concept:**

```solidity
contract MaliciousTrader is IERC1155Receiver {
    OrderBook public orderBook;
    uint256 public targetOrderId;
    bool public armed;

    function onERC1155Received(
        address, address, uint256, uint256, bytes calldata
    ) external returns (bytes4) {
        if (armed) {
            armed = false;
            orderBook.cancelOrder(targetOrderId); // Reenters OrderBook
        }
        return this.onERC1155Received.selector;
    }

    function attack(uint256 marketId, uint256 tick) external {
        // Place two bid orders
        uint256 orderId1 = orderBook.placeOrder(
            marketId, Side.Bid, OrderType.GoodTilBatch, tick, 10
        );
        uint256 orderId2 = orderBook.placeOrder(
            marketId, Side.Bid, OrderType.GoodTilBatch, tick, 10
        );
        targetOrderId = orderId2;
        armed = true;
        // When clearBatch settles orderId1, mintSingle callback fires,
        // cancels orderId2, and settlement reverts.
    }
}
```

**Recommended Fix:**

Add a settlement lock on OrderBook that prevents cancellations during batch clearing:

```solidity
// OrderBook.sol
bool public settlementActive;

modifier notDuringSettlement() {
    require(!settlementActive, "OrderBook: settlement in progress");
    _;
}

function setSettlementActive(bool active) external onlyRole(OPERATOR_ROLE) {
    settlementActive = active;
}

// Add notDuringSettlement to cancelOrder, cancelOrders, cancelExpiredOrder,
// cancelExpiredOrders, replaceOrders
```

Alternatively, defer ERC1155 transfers to a pull-based claim pattern, or batch all mints/transfers after settlement completes.

### M-01: PythResolver Confidence Check Bypassed When conf = 0

**Severity:** Medium
**Contract:** PythResolver.sol
**Function:** `_checkConfidence` (line 263)

**Description:**

```solidity
function _checkConfidence(int64 price, uint64 conf) internal view {
    if (conf == 0) return; // ← Bypasses all checks
    uint256 absPrice = price >= 0 ? uint256(uint64(price)) : uint256(uint64(-price));
    uint256 maxConf = (absPrice * confThresholdBps) / 10000;
    require(uint256(conf) <= maxConf, "PythResolver: confidence too wide");
}
```

If Pyth returns `conf = 0`, the confidence check is entirely skipped. While Pyth typically returns non-zero confidence for active feeds, this creates a bypass path. An attacker who can manipulate or find Pyth update data with `conf = 0` could submit resolution with an arbitrarily unreliable price.

Additionally, for very small prices (`absPrice * confThresholdBps < 10000`), `maxConf` rounds to 0, making it impossible to resolve markets for low-price assets unless `conf = 0`.

**Impact:** Potential oracle manipulation if Pyth ever returns zero confidence, or inability to resolve markets for very low-price assets.

**Recommended Fix:**

```solidity
function _checkConfidence(int64 price, uint64 conf) internal view {
    uint256 absPrice = price >= 0 ? uint256(uint64(price)) : uint256(uint64(-price));
    require(absPrice > 0, "PythResolver: zero price");
    uint256 maxConf = (absPrice * confThresholdBps) / 10000;
    if (maxConf == 0) maxConf = 1; // Minimum confidence band
    require(uint256(conf) <= maxConf, "PythResolver: confidence too wide");
}
```

### M-02: Redemption.redeem Silently Truncates Amount for Internal Positions

**Severity:** Medium
**Contract:** Redemption.sol
**Function:** `redeem` (line 73)

**Description:**

```solidity
function redeem(uint256 factoryMarketId, uint256 amount) external nonReentrant {
    // ...
    if (useInternalPositions) {
        vault.redeemPosition(msg.sender, orderBookMarketId, uint128(amount), outcomeYes);
        //                                                   ^^^^^^^^^^^^^^^^
        // Silent truncation if amount > type(uint128).max
    } else {
        outcomeToken.redeem(msg.sender, orderBookMarketId, amount, outcomeYes);
        uint256 payout = amount * LOT_SIZE;
        vault.redeemFromPool(orderBookMarketId, msg.sender, payout);
    }
}
```

The `amount` parameter is `uint256` but is cast to `uint128` for the internal positions path. If `amount > type(uint128).max`, it silently truncates. While positions are stored as `uint128` (making overflow practically impossible), the truncated value could still be less than or equal to the user's position, causing them to redeem fewer tokens than intended without any error.

The ERC1155 path uses the full `uint256` amount, creating an inconsistency.

**Impact:** Users on internal position markets could silently redeem fewer tokens than intended if they pass a large amount value.

**Recommended Fix:**

```solidity
if (useInternalPositions) {
    require(amount <= type(uint128).max, "Redemption: amount overflow");
    vault.redeemPosition(msg.sender, orderBookMarketId, uint128(amount), outcomeYes);
}
```

### M-03: Chunked Settlement Re-Computes Fills With Stale State, Causing Rounding Over-Fill

**Severity:** Medium
**Contract:** BatchAuction.sol
**Function:** `_settleChunk`, `_computeFills`

**Description:**

When a batch exceeds `SETTLE_CHUNK_SIZE` (400 orders, possible via uncapped GTC rollover), settlement is split across multiple `clearBatch` calls. Each call to `_settleChunk` re-invokes `_computeFills` for ALL orders in the batch, not just the current chunk.

After chunk 1 settles, it calls `orderBook.reduceOrderLots()` on settled orders, reducing their `lots` in storage. When chunk 2 calls `_computeFills`, it re-reads these modified lots. While the base pro-rata fill for unsettled orders is unaffected (because `BatchResult.totalBidLots/totalAskLots` are immutable), the **rounding remainder distribution changes**.

In `_computeFills`, the rounding remainder (`matchedLots - totalFilledSide`) is distributed +1 per order walking backwards. With reduced lots from settled orders, `totalFilledSide` is lower, making the remainder larger. This expanded remainder can grant +1 to orders that should NOT have received it in the correct global computation.

**Concrete Example:**

- 500 bid orders (10 lots each), 200 ask orders (10 lots each). totalBid=5000, matched=2000.
- Global: each bid gets `floor(10 * 2000/5000)` = 4 lots. Remainder = 0. No +1 adjustments needed.
- Chunk 1 settles bids 0-399 with 4 lots each. Their storage lots reduce to 6.
- Chunk 2 re-computes: bids 0-399 now have lots=6, fill = `floor(6 * 2000/5000)` = 2. `totalFilledBid = 400*2 + 100*4 = 1200`. New remainder = `2000 - 1200 = 800`.
- Backward +1 distribution gives bids 400-499 fill = `4 + 1 = 5` instead of correct `4`.
- Result: 100 orders each over-filled by 1 lot = 100 extra outcome tokens minted.

**Impact:**

Over-minting of outcome tokens by up to 1 lot ($0.01) per affected order in chunks after the first. Causes micro-insolvency of the market pool at redemption. Requires batches > 400 orders (only via GTC rollover accumulation). Impact scales linearly with excess orders.

**Recommended Fix:**

Option A — Store pre-computed fills and reuse across chunks:

```solidity
mapping(uint256 => mapping(uint256 => uint256[])) public storedFills;

// In first clearBatch call, compute and store all fills.
// In subsequent chunk calls, read from storage instead of re-computing.
```

Option B — Cap GTC rollover to prevent multi-chunk batches:

```solidity
function pushBatchOrderId(...) external onlyRole(OPERATOR_ROLE) returns (bool) {
    require(
        batchOrderIds[marketId][batchId].length < MAX_ORDERS_PER_BATCH,
        "OrderBook: batch full"
    );
    batchOrderIds[marketId][batchId].push(orderId);
    return true;
}
```

### L-01: No Upper Bound on GTC Rollover Batch Size

**Severity:** Low
**Contract:** OrderBook.sol
**Function:** `pushBatchOrderId` (line 496)

**Description:**

While new order placement is capped at `MAX_ORDERS_PER_BATCH` (400), GTC rollover via `pushBatchOrderId` has no limit. Over many batches with low fill rates, a batch can accumulate thousands of orders, causing increasing gas costs for `_computeFills` and the rounding issue in M-03.

**Recommended Fix:** Add a cap to `pushBatchOrderId`, auto-cancelling GTC orders that can't roll over.

### L-02: Sell Orders Pay Zero Protocol Fees

**Severity:** Low
**Contract:** OrderBook.sol, BatchAuction.sol

**Description:**

Buy orders (Bid/Ask) pay a 20 bps fee on their locked collateral. Sell orders (SellYes/SellNo) lock tokens, not USDT, and pay no fee at placement or settlement. In `_settleSellOrder`, the full payout at clearing price is sent to the seller with no fee deduction.

This creates an asymmetry where secondary market sellers avoid protocol fees entirely.

**Impact:** Revenue leakage for the protocol. May be intentional design.

**Recommended Fix:** If fees on sell orders are desired, deduct a fee from the sell payout before transferring.

### L-03: No Batch Interval Enforcement in clearBatch

**Severity:** Low
**Contract:** BatchAuction.sol
**Function:** `clearBatch`

**Description:**

The `Market` struct stores `batchInterval` (minimum seconds between batch auctions), but `clearBatch` does not enforce any minimum time between clears. Batches can be cleared in consecutive blocks. The field is stored but never checked.

**Impact:** MEV bots could rapidly clear batches, though FBA design largely mitigates this. The unused field could confuse integrators.

**Recommended Fix:** Either enforce the interval or remove the field.

### I-01: PythResolver Challenge Mechanism Favors First Resolver

**Severity:** Informational
**Contract:** PythResolver.sol

The finality timer (`resolvedAtTimestamp`) is set once on first submission and never reset on challenge. A challenge submitted at second 89 leaves only 1 second for counter-challenges. This is documented as intentional (prevents indefinite delay griefing), but the first resolver has a timing advantage.

---

### I-02: Low-Price Assets May Fail Confidence Check

**Severity:** Informational
**Contract:** PythResolver.sol

For assets with very low Pyth prices, `maxConf = (absPrice * confThresholdBps) / 10000` rounds to 0, making it impossible to resolve with any `conf > 0`. The admin can adjust `confThresholdBps`, but the minimum is 10 bps.

---

### I-03: Internal Position uint128 Arithmetic Safety

**Severity:** Informational
**Contract:** Vault.sol

Position tracking uses `uint128` for lots. While Solidity 0.8's checked arithmetic prevents silent overflow, `creditPosition` could theoretically revert at extreme accumulation. At $0.01/lot, this requires > $3.4 × 10^36. No action needed.

## Settlement Math Verification

**Buy order excess refund (BatchAuction._settleAmounts L312):**

```
excessRefund = (lockedForFilled + lockedFeeForFilled) - filledCollateral - protocolFee
```

- For Bid: orderTick >= clearingTick (participation condition), so lockedForFilled >= filledCollateral. Fee is proportional, so lockedFeeForFilled >= protocolFee. **No underflow.** ✓
- For Ask: orderTick <= clearingTick, so (100-orderTick) >= (100-clearingTick), so lockedForFilled >= filledCollateral. **No underflow.** ✓

**Clearing tick computation (SegmentTree.findClearingTick):**

Binary search finds highest tick where cumBid >= cumAsk. The tick+1 correction (L167-186) correctly handles the edge case where matched volume is maximized at clearingTick+1. **Correct.** ✓

**Pro-rata fill computation (_calcFilledLots):**

`fill = (lots * matchedLots) / totalSideLots` when oversubscribed. Each order gets at most `lots` (since matchedLots < totalSideLots). Rounding remainder adds at most +1 per order, which cannot exceed `lots` because `floor(lots * matchedLots / totalSideLots) + 1 <= lots` when `totalSideLots > matchedLots`. **Correct.** ✓

**Pool conservation:**

- Bid+Ask match at tick t: pool += LOT_SIZE (t/100 from Bid + (100-t)/100 from Ask)
- Bid+SellYes match at tick t: pool += t/100 (from Bid) - t/100 (to SellYes) = 0
- SellNo+Ask match at tick t: pool += (100-t)/100 (from Ask) - (100-t)/100 (to SellNo) = 0
- Redemption: pool -= LOT_SIZE per winning token

Pool balance >= outstanding winning tokens × LOT_SIZE at all times (assuming no chunked over-fill from M-03). **Correct.** ✓
