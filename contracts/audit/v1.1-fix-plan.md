**Date:** 2026-03-22
**Based on:** v1.1-audit-report.md
**Status:** PENDING REVIEW

## Fix 1: GTB Zero-Fill Order Cleanup (M-01)

**Problem:** GTB orders that cross the clearing tick but receive 0 fill from pro-rata rounding become orphaned — lots remain in storage, volume stays in SegmentTree, collateral/tokens stay locked. Users must manually `cancelOrder()` to recover.

**Fix:**
In `BatchAuction._settleOrder()`, add GTB cleanup to the `precomputedFill == 0` path:
- Call `reduceOrderLots(orderId, o.lots)` to zero out the order
- Call `updateTreeVolume()` to remove phantom volume from SegmentTree
- Return collateral (buy orders) or tokens (sell orders) to the owner
- For internal positions, call `vault.unlockPosition()`

**Files changed:**
- `src/BatchAuction.sol` — `_settleOrder()` zero-fill path

**Tests:**
- Add test: GTB bid at clearing tick, many competing orders, pro-rata rounds to 0 → verify order cleaned up, tree volume reduced, collateral returned
- Add test: GTB sell order (SellYes/SellNo) same scenario → verify tokens returned
- Add test: GTB with internal positions same scenario → verify positions unlocked

## Fix 2: Chunked Settlement Correctness (L-01)

**Problem:** When settlement spans multiple chunks, `_computeFills` recomputes from scratch in chunk 2+, but chunk 1 already mutated order lots via `reduceOrderLots`. GTC partial-fill orders now have fewer lots than original, causing incorrect pro-rata fills and potential reverts.

**Fix:**
Store precomputed fills during the first chunk, read them in subsequent chunks:
- Add `mapping(uint256 => uint256) private _precomputedFills` (orderId → fill lots)
- First chunk: `_computeFills` → write results to mapping
- Subsequent chunks: read from `_precomputedFills` instead of recomputing
- Clear the mapping entries after the final chunk completes (or use batchId-scoped storage)

**Files changed:**
- `src/BatchAuction.sol` — `_settleChunk()`, new `_precomputedFills` mapping, cleanup logic

**Tests:**
- Add test: batch with > SETTLE_CHUNK_SIZE orders (requires GTC rollovers to exceed 400), verify all orders settle correctly across chunks
- Add test: GTC partial-fill order in chunk 1, remaining lots correct in chunk 2
- Verify no storage leaks — precomputed fills cleared after final chunk

## Fix 3: Raise MAX_ORDERS_PER_BATCH (L-01+)

**Problem:** `MAX_ORDERS_PER_BATCH == SETTLE_CHUNK_SIZE == 400`. Chunked settlement never activates, and GTC rollovers can fill up batches quickly, pushing new orders out.

**Fix:**
- Raise `MAX_ORDERS_PER_BATCH` to 1600 (4 × SETTLE_CHUNK_SIZE)
- Keep `SETTLE_CHUNK_SIZE` at 400
- This gives chunked settlement room to operate and provides more headroom for resting orders

**Files changed:**
- `src/BatchAuction.sol` or `src/OrderBook.sol` — wherever `MAX_ORDERS_PER_BATCH` is defined

**Tests:**
- Update any tests that assert on `MAX_ORDERS_PER_BATCH` value
- Existing chunked settlement tests (Fix 2) implicitly validate this

## Fix 4: Price-Proximity Batch Filtering + Per-User Order Cap (NEW — DoS Prevention)

**Problem:** An attacker can cheaply place hundreds/thousands of GTC orders at extreme prices (e.g. tick=1 when mid=50) for ~$0.16 total, permanently clogging `batchOrderIds` and burning gas on settlement.

### Fix 4a: Price-Proximity Filtering

When a batch is full or an order is far from the reference price, park it in a resting list instead of the active batch.

**Design:**
- `mapping(uint256 => uint256[]) restingOrderIds` — parked orders per market
- `mapping(uint256 => uint8) lastClearingTick` — reference price per market (updated after each clear)
- Define proximity threshold (e.g. 20 ticks from lastClearingTick)
- Resting orders are **NOT in the SegmentTree** — no phantom volume, no price distortion
- On `placeOrder`: if order tick is outside proximity range AND batch is near capacity, push to `restingOrderIds` instead of `batchOrderIds`, do NOT add to tree
- On `clearBatch` (before computing): scan `restingOrderIds`, pull any orders now within range into `batchOrderIds` and add their volume to the tree
- On `cancelOrder`: if order is in resting list, remove it (lazy-skip or explicit removal)
- Emit `OrderResting(orderId)` event (distinct from `OrderPlaced`) so frontend can show "parked, activates when price approaches"
- On `_tryRollOrCancel` for GTC: if order is now far from lastClearingTick, move to resting list instead of rolling into next batch

**Gas-bounded scan:**
- The pull-in scan in `clearBatch` MUST be gas-bounded — cap at e.g. `MAX_RESTING_PULL = 200` per clear
- If more than 200 resting orders are now in range, process the rest in subsequent `clearBatch` calls
- Track a `restingScanIndex` per market to resume where we left off
- This prevents OOG if price jumps suddenly and thousands of resting orders become active at once

**Edge cases to handle:**
- First batch (no lastClearingTick yet): use 50 as default midpoint, or don't filter until first clear
- Cancelled resting orders: lazy-skip during scan (check `o.lots == 0`) — cheapest approach
- Market with no activity: resting orders stay parked until price action arrives
- Large price move: may take multiple batches to pull in all newly-in-range orders (acceptable — 3s per batch)

**Files changed:**
- `src/BatchAuction.sol` — resting list management, pull-in logic at start of `clearBatch`, `_tryRollOrCancel` modification
- `src/OrderBook.sol` — `placeOrder` routing logic (active vs resting), `cancelOrder` handling for resting orders, `OrderResting` event
- `src/ITypes.sol` — add `OrderResting` event if events are defined here

### Fix 4b: Per-User Active Order Cap

Simple rate limit: max 20 active orders per user per market.

**Design:**
- `mapping(address => mapping(uint256 => uint16)) activeOrderCount`
- Increment on `placeOrder` (both active and resting orders count toward cap)
- Decrement on cancel, fill (full), expire
- `require(activeOrderCount[msg.sender][marketId] < MAX_USER_ORDERS, "too many orders")`
- `MAX_USER_ORDERS = 20` (constant)

**Files changed:**
- `src/OrderBook.sol` — counter mapping, increment in `placeOrder`/`placeOrders`, decrement in `_cancelCore`/settlement callbacks
- `src/BatchAuction.sol` — decrement on full fill settlement

**Tests:**
- Test: place 20 orders → 21st reverts
- Test: cancel one → can place again
- Test: order fills → count decremented → can place new
- Test: order at extreme tick goes to resting list, collateral locked, cancel returns it
- Test: price moves to resting order range → pulled into batch → fills correctly
- Test: `OrderResting` event emitted for parked orders
- Test: resting order cancelled → removed from resting list (or lazy-skipped)
- Test: GTC order rolls from active to resting when price moves away
- Test: attacker fills 20 orders × multiple markets → only affects their own markets

## Fix 5: Split Protocol Fees 50/50 (I-01)

**Problem:** Buy orders pay full protocol fee (20bps). Sell orders pay zero. This creates asymmetric fee treatment on secondary market trades.

**Fix:**
Split the fee equally: buyer pays 10bps, seller pays 10bps (total 20bps maintained).

**Design:**
- In `_settleSellOrder()`: compute sell fee as `feeModel.calculateFee(sellPayout) / 2` or introduce a `calculateHalfFee` method
- Deduct sell fee from USDT payout before sending to seller
- In `_settleAmounts()` for buy side: use half fee instead of full fee
- Alternative (cleaner): `FeeModel.calculateFee()` returns the per-side fee (half of total), both sides call it once
- Protocol fee collection remains in Vault, just split between both settlement paths

**Edge case:** Odd-number fee amounts (1 wei rounding). Give the extra wei to protocol (round up on one side).

**Files changed:**
- `src/BatchAuction.sol` — `_settleSellOrder()`, `_settleAmounts()`
- `src/FeeModel.sol` — possibly add `calculateHalfFee()` or adjust `calculateFee()` semantics
- `src/Vault.sol` — if fee accounting needs updating

**Tests:**
- Test: buy and sell in same batch → both pay equal fees
- Test: verify total fees collected = same as before (20bps of matched volume)
- Test: odd collateral amount → rounding goes to protocol
- Test: pool solvency maintained with split fees (critical — update fuzz tests)
- Update existing sell order tests that assert zero seller fees

## Fix 6: Document MEV Exposure (L-03)

**Problem:** `batchInterval` stored but not enforced. Permissionless `clearBatch` enables MEV sandwich attacks that isolate users in empty batches.

**Fix:** Documentation only — enforcing minimum batch duration adds latency.

**Files changed:**
- `src/BatchAuction.sol` — add NatSpec comments on `clearBatch` explaining:
  - Why batchInterval is not enforced (latency tradeoff)
  - The MEV attack vector (front-run clear → isolate → match at limit)
  - Proposed future mitigation: enforce `block.timestamp >= lastClear + batchInterval`
  - Note: price-proximity filtering (Fix 4) reduces this surface by keeping fewer orders exposed

## Implementation Order

1. **Fix 4b** (per-user cap) — simplest, immediate DoS protection
2. **Fix 1** (M-01 GTB cleanup) — straightforward, isolated change
3. **Fix 5** (fee split) — isolated to settlement logic
4. **Fix 3** (raise batch limit) — one constant change
5. **Fix 2** (chunked settlement) — needed before Fix 4a works safely
6. **Fix 4a** (price-proximity filtering) — largest change, depends on Fix 2 + Fix 3
7. **Fix 6** (MEV documentation) — last, no code changes

## Final Review Notes

**Risks to watch:**
- Fix 4a (resting list) is the most complex change — the scan loop in `clearBatch` must be gas-bounded (limit how many resting orders we pull per clear, iterate in subsequent clears if needed)
- Fix 5 (fee split) touches pool solvency invariant — fuzz tests are critical, verify `marketPool >= winningTokens × LOT_SIZE` still holds with split fees
- Fix 2 (precomputed fills) adds storage writes — gas cost increases for the first chunk, but only when chunking activates (batches > 400)

**Not addressed (by design):**
- L-02 (uint128 truncation) — non-issue in practice
- I-02 (late challenge timing) — acceptable tradeoff
- I-03 (emergency withdraw phantom positions) — cosmetic
- I-04 (fee cap) — fine for now
- I-05 (ERC1155 cancel revert) — not using ERC1155 currently
